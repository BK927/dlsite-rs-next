### 3) `product_id -> thumbnail`
Status: **Partially verified / narrow exception only**

Recommended approach:
- Keep this feature **opt-in** and explicitly marked as a narrow fallback.
- Only support extraction when the product page exposes clickable image anchors that resolve directly to `img.dlsite.jp` assets.
- Return `image_asset_url` unless you have positively verified that the asset is the canonical cover/thumbnail.

What was verified live:
- On a product page, clicking linked `Image:` elements can resolve directly to `img.dlsite.jp` image assets.
- For `RJ239066`, a clickable image resolves to `.../RJ239066_img_main_240x240.jpg`, which is clearly a canonical thumbnail-sized main image asset.
- For the game work `RJ01558964`, clickable `Image:` elements resolve to `img.dlsite.jp/modpub/images2/parts/...jpg` assets, but the verified examples are sample/detail images rather than a clearly identified canonical thumbnail.

Interpretation:
- The mechanism "product page -> direct image asset URL" is real.
- However, for game works, this live pass did **not** fully verify a uniform, canonical `thumbnail_url` extraction rule.
- So the safe implementation is:
  - support `image_asset_url` extraction,
  - do **not** promise canonical thumbnail quality for every game,
  - keep this outside the strict route-first core.

Operational guidance:
- Core client: no thumbnail dependency.
- Optional module: image-asset harvesting from verified clickable image anchors only.
- Distinguish:
  - `thumbnail_url` (only when confidently canonical)
  - `image_asset_url` (generic verified image asset)
- For now, game thumbnails should be considered **best-effort**, not guaranteed.

Why this only partially fits the philosophy:
- It touches product detail pages.
- It can still be acceptable as a very narrow exception because the extraction target is a direct asset URL, not arbitrary visual metadata.
- But it should not be treated as a hard dependency for the core client.

## Minimal implementation shape

```text
resolve_circle_name(circle_name)
  -> normalize(circle_name)
  -> choose bucket from first normalized char
  -> fetch /circle/list/=/name_header/{bucket}
  -> exact-match circle row
  -> return maker_id

list_circle_games(maker_id)
  -> fetch /circle/profile/=/maker_id/{maker_id}.html
  -> if total_count > current_page_size, follow per_page/100 or paginate
  -> parse works list cards
  -> filter by game-family work-type labels
  -> return normalized works

get_product_image_asset(product_id)
  -> fetch /work/=/product_id/{product_id}.html
  -> extract verified clickable image anchors only
  -> if canonical main-image pattern is confirmed, return thumbnail_url
  -> else return image_asset_url or null
```
